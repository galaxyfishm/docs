    # 一、关系型数据库特点

    • 基于行存储数据，二维的模式；
    • 存储结构化的数据，数据存储有固定模式（schema）；
    • 表与表之间存在关联；
    • 大都支持SQL（结构化查询语言）的操作，支持复杂的关联查询；
    • 通过支持事务ACID来提供严格或实时的数据一致性；

    缺点：

    • 要实现扩容的话，只能向上（垂直）扩容，不支持动态扩缩容；
    • 表结构修改困难，因此存储的数据格式也受到限制；
    • 高并发情况下，基于磁盘的读写压力比较大；

    # DML、DDL、DCL区别

    **DML(data manipulation language)：**它们是SELECT、UPDATE、INSERT、DELETE，就象它的名字一样，这4条命令是用来对数据库里的数据进行操作的语言。

    **DDL(data definition language)：**DDL比DML要多，主要的命令有CREATE、ALTER、DROP等，DDL主要是用在定义或改变表(TABLE)的结构，数据类型，表之间的链接和约束等初始化工作上，

    ​ 他们大多在建立表时使用。

    **DCL(Data Control Language)：**是数据库控制功能。是用来设置或更改数据库用户或角色权限的语句，包括(grant,deny,revoke等)语句。在默认状态下，只有sysadmin,dbcreator,db_owner或

    ​ db_securityadmin 等人员才有权力执行DCL。

    # 二、初识MySQL

    MySQL服务端默认最大连接数151。

    客户端与服务端连接：

    • 通信类型：同步、异步；
    • 连接方式：长连接、短连接；
    • 协议：TCP、Unix Socket；

    # 1. MySQL架构

    # 1. 架构分层

    MySQL可以分为三层，连接层（跟客户对接）、服务端（执行操作）、存储引擎（提供数据支持）。

    # 2. 执行查询语句

    # 1.1 连接

    MySQL服务默认监听端口是3306，**客户端与服务端连接方式有很多：**通信类型：同步、异步；连接方式：长连接、短连接；协议：TCP、Unix Socket；

    客户端每产生一个连接或者会话，在服务端就会创建一个线程来处理。反过来如果要杀死会话，就是kill线程。MySQL会把长时间不活动的连接自动断开。

    MySQL允许的最大连接数：默认151，最大可以设置为10000。

    # 1.4 查询优化器

    一条SQL语句可以有很多中执行方式，如何选择一条最优执行，这就是查询优化器所做的工作。

    查询优化器根据解析树生成不同的执行计划，然后选择一种最优的执行计划，MySQL使用基于开销的优化器，选择执行计划开销最小的。查看命令开销：

    • 子查询优化；
    • 等价谓词重写；

    • 嵌套连接消除；
    • 连接的消除；
    • 语义优化；
    • 非SPJ优化；

    优化器最终会把解析树变成一个查询执行计划，即一个数据结构。

    # 1.5 执行引擎

    执行引擎处理执行计划，调用存储引擎提供的统一API，最后把数据返回给客户端。如果开启缓存，同时把写入缓存。

    # 1.6 存储引擎

    存储引擎类型：InnoDB、Memory、MyISAM等，并且可以自己实现。

    各种存储引擎提供了统一的API，使得执行引擎调用变得简单，所以对于表的存储引擎类型可以进行修改。

    **MyISAM特点：**表级锁限制了读写性能，常用于以只读为主的工作。

    • 表级别锁，不支持事务。
    • 拥有较高插入和查询速度。
    • 存储了表的行数（count更快）。

    **InnoDB特点：**事务安全的引擎，支持行级锁，将数据存储在聚集索引中，还支持外键引用完整性约束。

    • 支持事务，支持外键，因此数据的完整性、一致性更高。
    • 支持行级别的锁和表级别的锁。
    • 支持读写并发，写不阻塞读（MVCC）。
    • 特殊索引存放方式，可以减少IO，提升查询效率。

    **Memory特点：**将所有数据存放在RAM中，也被称为堆引擎。

    • 读写速度很快，但是数据库重启或者奔溃，数据会消失。只适合做临时表。
    • 将数据存储在内存中。

    # 3. 执行更新语句

    **两阶段提交原因：**两阶段提交类似于分布式事务，如果不用两阶段无法保证redo log 和 bin log 的事务一致性。

    在崩溃时，判断事务是否需要提交：

    • bin log 无记录，redo log 无记录：在redo log 写之前crash，恢复操作-》回滚事务；
    • bin log 无记录，redo log 状态为 prepare：在bin log 写完之前crash，恢复操作-》回滚事务；
    • bin log 有记录，redo log 状态为 prepare：在bin log 写完提交事务之前的crash，恢复操作-》提交事务；
    • bin log 有记录，redo log 状态为 commit：正常完成事务，不需要恢复；

    # 4. InnoDB架构

    # 1. 内存结构

    # Buffer Pool

    Buffer Pool缓存的是页面信息，包括数据页、索引页。Buffer Pool默认大小128M，可以调整。show variables like '%innodb_buffer_pool%';

    InnoDB用LRU算法管理缓冲池（链表实现，非传统LRU，分young和old区）。

    预读机制（read ahead）：InnoDB的数据页并不都是在访问的时候才缓存到buffer pool，而是使用预读机制

    • 分为线性预读和随机预读；

    • 这种预读机制，将可能用到的数据提前加载到buffer pool，提高了I/O性能，但也会导致内存空间占用更多；

    • 更有效的清理缓存，LRU算法将list分为new sublist（5/8），old sublist（3/8）。即对buffer pool做冷热分离；

      • 所有新数据加入到buffer pool，一律放在LRU List的old sublist的head；
      • old sublist的数据再次被访问后（innodb_old_blocks_time=1 ，1秒以后被访问数据才移动，防止造成缓存污染），移动到new sublist；
      • new sublist的数据长时间没被访问，移动到old sublist。访问new sublist后3/4区域的缓存页，会移动到前1/4区域；
      • 将old sublist区tail的数据进行淘汰；

    # Change Buffer

    引入Buffer Pool后，在更新数据时，如果数据页在Buffer Pool中存在，那么直接更新。如果不存在，那是否意味着需要从磁盘加载到内存，再对内存的数据页进行操作？

    由上述问题，引入了Change Buffer，提高了更新时的性能。Change Buffer是Buffer Pool一部分。

    如果数据页不是唯一索引，不需要从磁盘加载索引页判断是不是重复（唯一性检查）。这种情况下可以先把（Insert、Update、Delete）记录在Buffer Pool，最后再记录到数据页（merge操作）。

    **何时merge：**在访问这个数据页的时候、后台线程、数据库shut down、redo log写满时触发。

    # Adaptive Hash Index

    # (Redo) Log Buffer

    Redo Log并不是每次直接写入磁盘，而是在Buffer Pool里有一块内存区域（Log Buffer）专门用来记录即将要写入日志文件的数据，默认16M，可以节省磁盘I/O。

    查看Log Buffer写入磁盘时间，show variables like 'innodb_flush_log_at_trx_commit'; 刷盘越快，越安全，但会消耗性能。

    # 2. 磁盘结构

    Tables、Indexes、Tablespaces、InnoDB Data Dictionary、Doublewrite Buffer、Redo Log、Undo Logs

    System Tablespace：InnoDB数据字典（存储表和索引的元数据）、双写缓冲、change buffer、undo logs，如果没有指定file-per-table，也包含用户创建的表和索引数据。

    部分写失效，可以使用双写缓冲解决此问题。

    # 5. 后台线程

    后台线程主要作用是负责刷新内存池中的数据和把修改的数据页刷新到磁盘。后台线程分为：master thread、IO thread、purge thread、page cleaner thread。

    master thread：负责刷新缓存数据到磁盘并协调调度其他后台进程。

    IO thread：分为insert buffer、log、read、write进程。分别用来处理insert buffer、重做日志、读写请求的IO回调。

    purge thread：用来回收undo页。

    page cleaner thread：用来刷脏。

    # 三、MySQL事务

    事务：是数据库管理系统（DBMS）执行过程中的一个逻辑单位，由一个有限的数据库操作序列构成。单条（DDL：create、drop）和（DCL：grant、revoke）也会有事务。

    MySQL中支持事务的存储引擎：NDB、InnoDB。

    事务四大特性（存储引擎：InnoDB）：

    • 原子性（Atomicity）：通过 undo log 实现原子性；
    • 隔离性（Isolation）：普通 select 使用 MVCC，加锁 select 使用记录锁、间隙锁、临键锁。
    • 持久性（Durability）：通过 redo log 和 double write buffer 来实现；
    • 一致性（Consistent）：

    # 1. 事务并发问题

    # 脏读

    # 不可重复读

    # 幻读

    不可重复读和幻读区别：修改或者删除造成读不一致叫不可重复读，插入造成的读不一致叫幻读。

    # 2. 事务隔离级别（SQL92标准）

    P1：脏读；P2：不可重复读；P3：幻读；

    READ UNCOMMITTED（读未提交）：会出现脏读、不可重复读、幻读；

    READ COMMITTED（已提交读）：会出现不可重复读、幻读；

    REPEATABLE READ（可重复读）：会出现幻读；

    SERIALIZABLE（串行化）：解决所有问题；

    # 1. MySQL InnoDB隔离级别

    # LBCC

    解决读一致性两大方案之一。Lock Based Concurrency Control（LBCC），读取数据时锁定要操作的数据，不允许其他的事务进行修改。不支持并发读写，读多写小情况下非常影响性能。

    # MVCC

    解决读一致性两大方案之一。Multi Version Concurrency Control（MVCC），在修改数据之前建立一个快照，后续事务中只会读取这个快照。

    MVCC原则：

    • 一个事务能看到的数据版本：
      • 第一次查询之前已经提交的事务的修改；
      • 本事务的修改；
    • 一个事务不能看见的数据版本：
      • 本事务第一次查询之后创建的事务（事务ID比我的事务ID大）；
      • 活跃的（未提交的）事务修改；

    MVCC原理：

    InnoDB的事务都是有编号的，而且会不断递增。InnoDB为每行记录都实现了两个隐藏字段：DB_TRX_ID（事务ID）、DB_ROLL_PTR（回滚指针）。

    Read View（可见性视图）：

    根据 Read View，事务判断可见性规则：

    • 从数据的最早版本开始判断（undo log）；

    • trx_id = creator_trx_id，本事务修改，可以访问；

    • trx_id < min_trx_id，说明这个版本在生成 ReadView 已经提交，可以访问；

    • trx_id > max_trx_id，说明这个版本是生成 ReadView 之后才开启的事务，不能访问；

    • min_trx_id < trx_id < max_trx_id，看是否在 m_ids 中，如果在不能访问；如果不在可以访问；

    • 如果当前版本不可见，就找 undo log 链中的下一个版本；

    RR 中 Read View 是事务第一次查询的时候建立。RC 中 Read View 是事务每次查询的时候建立的（无法解决幻读）。

    # 3. InnoDB锁类型

    • Shared and Exclusive Locks
    • Intention Locks
    • Record Locks
    • Gap Locks
    • Next-Key Locks
    • Insert Intention Locks
    • AUTO-INC Locks
    • Predicate Locks for Spatial Indexes

    行级别锁（Shared and Exclusive Locks）和表级别锁（Intention Locks），称为锁的基本模型。

    Record Locks、Gap Locks、Next-Key Locks，称为锁的算法。

    Insert Intention Locks，插入意向锁。

    AUTO-INC Locks，自增锁。是一种特殊的表锁，用来防止自增字段重复，数据插入以后就会释放，不需要等到事务提交才释放。show variables like 'innodb_autoinc_lock_mode';

    • 0：traditional（每次都会产生表锁）
    • 1：consecutive（会产生一个轻量锁，simple insert 会获得批量的锁，保证连续插入，默认值）
    • 2：interleaved（不会锁表，来一个处理一个，并发最高）

    Predicate Locks for Spatial Indexes，数据类型的索引。

    # 共享锁

    Shared Locks（共享锁），行级别的锁。获取了一行数据的读锁，注意不要在加上读锁以后去写数据，可能会出现死锁的情况。多个事务可以共享一把读锁。

    共享锁的作用：共享锁会阻塞其他事务的修改，所以可以用在不允许其他事物修改数据的情况。

    BEGIN; SELECT * from user where name = 'King' lock in share mode;

    # 排它锁

    Exclusive Locks（排它锁），行级别的锁。用来操作数据，所以又叫写锁。只要一个事务获取了一行数据的排它锁，其他事物就不能再获取这一行数据的共享锁和排它锁。

    增删改默认自动加排它锁，for update 手动加排它锁。

    # 意向锁

    意向锁由数据库自己维护。当我们给一行数据加上共享锁之前，数据库会自动在这张表上加上一个意向共享锁。当我们给一行数据加上排它锁之前，会自动为这张表加上一个意向排它锁。

    **意向锁的作用：**在需要加表锁时，不必要进行全表扫描，只需要判断表上是否存在意向锁，如果有直接返回失败。如果没有，就可以加锁成功。

    # 4. 行锁的原理

    InnoDB 的行锁，是通过锁住索引实现的。

    **一张表有没有可能没有索引：**①、如果定义了主键，那么 InnoDB 会选择主键作为聚集索引。

    ​ ②、如果没有显示定义主键，InnoDB 会选择一个不包含 NULL 值的唯一索引作为主键索引。

    ​ ③、如果没有这样的唯一索引，InnoDB 会选择内置6字节的 ROWID 作为隐藏的聚集索引。

    **案例一：**为什么没有索引的表，锁住一行数据会导致锁表？

    因为查询没有使用索引，会进行全表扫描，然后把每一个隐藏的聚集索引都锁住了。

    **案例二：**为什么通过唯一索引给数据加行锁，主键索引也会被锁住？

    因为唯一索引中存储的是主键的值，所以最终会根据主键的值去主键索引中搜索叶子节点存储的数据，进而导致主键索引也会被一同锁住。

    # 5. 锁的算法

    整型的主键索引，它是可以排序的，主键索引不是整型是字符情况下也是有相应的排序规则。

    Next-key Lock（临键锁） = Gap Lock（间隙锁） + Record Lock（记录锁）

    InnoDB 的 RR 级别能够解决幻读的问题，就是用临键锁实现的。

    # 6. 死锁

    当多个事务形成等待环路的时候，即发生死锁。

    查看锁日志信息（非实时）：SHOW STATUS LIKE 'innodb_row_lock%';

    • Innodb_row_lock_current_waits：当前正在等待锁定数量；
    • Innodb_row_lock_time：从系统启动到现在，锁定总时长，单位ms；
    • Innodb_row_lock_time_avg：每次等待所花平均时间；
    • Innodb_row_lock_time_max：从系统启动到现在，最长一次等待时间；
    • Innodb_row_lock_waits：从系统启动到现在，总共发生等待次数；

    当前运行的所有事务：SELECT * FROM information_schema.INNODB_TRX;

    如果一个事务长时间不释放锁，可以 kill 事务的线程ID。

    存储引擎运行状态：SHOW ENGINE INNODB STATUS;

    开启 InnoDB 监控：SET GLOBAL innodb_status_output=ON; SET GLOBAL innodb_status_output_locks=ON;

    # EXPLAIN

    id：id值不同，先大后小；id值相同，从上往下；

    select_type：SIMPLE-简单查询；PRIMARY-主查询；SUBQUERY-子查询；DERVIED-衍生查询，用到临时表；....

    type：system > const > eq_ref > ref > fulltext > ref_or_null > index_merge > unique_subquery > index_subquery > range > index > ALL 除了ALL 都能用索引。

    • const：主键索引或唯一索引，智能查到一条数据；
    • eq_ref：通常出现多表 join 查询，被驱动表通过唯一性索引查询，此时被驱动表就是eq_ref；
    • ref：用到了非唯一性索引，或者关联操作只使用了索引的最左前缀；
    • range：使用索引范围扫描；
    • index：查询全部索引中的数据；
    • all：没有用索引；

    key_len：索引的长度，跟索引类型、长度有关。

    filtered：存储引擎返回的数据在 server 层过滤后，剩下多少满足查询的记录数量的比例。

    Extra：执行计划给出的额外信息。

    • using index：用到了覆盖索引，不需要回表；
    • using where：使用了 where 过滤，表示存储引擎返回的数据并不是所有都满足，需要在 server 层进行过滤；
    • using index condition：索引条件下推；
    • using filesort：不能使用索引排序，用到了额外的排序，需要优化；
    • using temporary：用到临时表；（distinct、group by、join）

    # SQL与索引优化

    例如：SELECT * FROM user LIMIT 10000000, 10; -》 SELECT * FROM user WHERE id > 10000000 LIMIT 10; 先走 id 索引，再获取10条数据。
